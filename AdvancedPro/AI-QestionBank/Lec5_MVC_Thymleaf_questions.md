## Section 1 — Topic Summary

- **ResultSet Definition:** A `ResultSet` object is a database table representation generated by executing a database query using `Statement` or `PreparedStatement`. It provides access to the query results.
- **Statement & ResultSet Association:** Only one `ResultSet` object per `Statement` can be open at any given time. Executing a new query with the same `Statement` implicitly closes any existing `ResultSet` created by it.
- **Sequential Retrieval & Cursor:** Table rows in a `ResultSet` are retrieved sequentially. The `ResultSet` maintains a cursor that points to the current row of data.
- **Cursor Movement (`next()`):**
  - The cursor is initially positioned **before the first row** (index 0 / before-first).
  - The `next()` method moves the cursor forward to the next row.
  - The first call to `next()` activates the first row.
  - `next()` returns `true` if the new current row is valid, and `false` if there are no more rows left in the result set.
- **Resource Cleanup (`close()`):**
  - The `close()` method disposes of the `ResultSet` resources immediately.
  - Properly closing a `ResultSet` is crucial to prevent database cursor leaks and to allow the `Statement` that created it to be re-used.

## Multiple choice (MCQ)

### Q1 — Multiple choice | Bloom's: Remember | Difficulty: Easy | Marks: 2

**Question:**
When a JDBC `ResultSet` object is first returned from executing a query, where is its cursor positioned?

A. Pointing directly to the first row of the retrieved table.  
B. Pointing to the last row of the retrieved table.  
C. Positioned before the first row of the retrieved table.  
D. Positioned after the last row of the retrieved table.  

**Guide answer:**
C

**Key points to include for Full Marks:**
- The cursor is initially positioned before the first row.
- You must call `next()` to move to the first row before reading any column value.

**Common mistakes to avoid:**
Choosing option A, assuming that the data is ready to be read immediately without calling `next()`.

### Q2 — Multiple choice | Bloom's: Understand | Difficulty: Medium | Marks: 2

**Question:**
What happens if you execute a new database query using a `Statement` object while a `ResultSet` object previously returned by the same `Statement` is still open and being iterated?

A. The new query executes successfully, and both `ResultSet` objects remain open and active.  
B. The first `ResultSet` is automatically and implicitly closed.  
C. An `SQLException` is immediately thrown at runtime indicating multiple open result sets.  
D. The second query is blocked until the first `ResultSet` is manually closed.  

**Guide answer:**
B

**Key points to include for Full Marks:**
- By JDBC specification, only one `ResultSet` object per `Statement` can be open at the same time.
- Creating a new `ResultSet` on the same `Statement` closes the old one.

**Common mistakes to avoid:**
Believing that multiple result sets can coexist on a single raw JDBC `Statement` instance.

## Short answer

### Q5 — Short answer | Bloom's: Understand | Difficulty: Medium | Marks: 4

**Question:**
Explain what happens if a developer attempts to call `rs.getString("name")` on a newly created `ResultSet` object `rs` before calling `rs.next()`.

**Guide answer:**
An `SQLException` will be thrown at runtime. When a `ResultSet` is created, its cursor is positioned before the first row. Getter methods (such as `getString()`) require the cursor to be pointing to a valid row containing data. Since `next()` has not been called yet to advance the cursor to the first row, no row is active, resulting in an error.

**Key points to include for Full Marks:**
- State that an `SQLException` is thrown.
- Explain that the cursor is initially positioned *before the first row*.
- Highlight that getters require a valid active row, which is only set after calling `next()`.

**Common mistakes to avoid:**
Claiming it returns `null` or empty string instead of throwing an exception.

## True / False + justify

### Q6 — True / False + justify | Bloom's: Analyse | Difficulty: Medium | Marks: 3

**Question:**
**T/F:** It is safe to skip calling `ResultSet.close()` as long as the parent `Statement` and `Connection` objects are closed, because Java's Garbage Collector will automatically clean up the database cursors.

**Guide answer:**
**False.** Relying on the Garbage Collector to clean up database resources is unsafe and causes resource leaks. Databases have a limit on the number of open cursors. If result sets are not closed explicitly (or via try-with-resources), the cursors remain open on the database server, eventually leading to "maximum open cursors exceeded" errors.

**Key points to include for Full Marks:**
- State **False**.
- Explain that database cursors remain open on the server.
- Mention that this causes resource/cursor leaks and potential database connection starvation.

**Common mistakes to avoid:**
Assuming garbage collection manages database server-side resources instantly.

## Application (code / scenario-based)

### Q7 — Application | Bloom's: Apply / Create | Difficulty: Hard | Marks: 10

**Question:**
Write a complete, syntactically correct Java method `displayHighGrades(Connection conn)` that queries a database table named `Students` for all students with a grade of `A`. The method must:
1. Retrieve columns `student_id` (int) and `student_name` (String).
2. Sequentially print the results to the console in the format: `ID: [id] | Name: [name]`.
3. Handle any potential `SQLException`s.
4. Ensure all JDBC resources (`Statement` and `ResultSet`) are closed properly using the Java try-with-resources statement.

**Guide answer:**
```java
import java.sql.Connection;
import java.sql.Statement;
import java.sql.ResultSet;
import java.sql.SQLException;

public void displayHighGrades(Connection conn) {
    String sql = "SELECT student_id, student_name FROM Students WHERE grade = 'A'";
    
    // Using try-with-resources to automatically close Statement and ResultSet
    try (Statement stmt = conn.createStatement();
         ResultSet rs = stmt.executeQuery(sql)) {
        
        // Loop through the ResultSet sequentially
        while (rs.next()) {
            int id = rs.getInt("student_id");
            String name = rs.getString("student_name");
            System.out.println("ID: " + id + " | Name: " + name);
        }
        
    } catch (SQLException e) {
        System.err.println("Database error: " + e.getMessage());
    }
}
```

**Key points to include for Full Marks:**
- Correct imports (`java.sql.*`).
- Use of `try-with-resources` to declare both `Statement` and `ResultSet` to ensure automated cleanup.
- A `while (rs.next())` loop to traverse the records sequentially.
- Correct retrieval methods: `rs.getInt("student_id")` and `rs.getString("student_name")`.
- Catching `SQLException`.

**Common mistakes to avoid:**
- Forgetting to import sql classes.
- Accessing data without calling `rs.next()`.
- Closing resources manually in a way that might skip execution if an exception is thrown (e.g., not placing them in a `finally` block or try-with-resources).
- Using `if(rs.next())` instead of `while(rs.next())` which only retrieves the first row instead of all rows.

## Compare & contrast

### Q8 — Compare & contrast | Bloom's: Analyse | Difficulty: Medium | Marks: 6

**Question:**
Compare the behavior, return type, and purpose of `ResultSet.next()` versus `ResultSet.close()`.

**Guide answer:**

| Feature | `ResultSet.next()` | `ResultSet.close()` |
| :--- | :--- | :--- |
| **Return Type** | `boolean` | `void` |
| **Purpose** | Moves the cursor forward to the next row and checks if a row exists. | Disposes of the `ResultSet` and releases database/JDBC resources immediately. |
| **State Change** | Modifies the cursor position (advances by 1 row). | Closes the result set, making it unusable for further reads. |
| **Resource Effect** | None. Keeps database connection and cursors open. | Releases database cursors and enables reuse of the parent `Statement`. |

**Key points to include for Full Marks:**
- Explicit comparison of return types (`boolean` vs `void`).
- Contrast in purpose (cursor traversal vs resource disposal).
- Mention of the effect on resources (keeps open vs closes).

**Common mistakes to avoid:**
Forgetting to specify the return types or their different impacts on parent Statement reusability.

## Essay / long answer

### Q9 — Essay / long answer | Bloom's: Evaluate | Difficulty: Hard | Marks: 12

**Question:**
Discuss the cursor-based model of a JDBC `ResultSet`. Explain the sequence of steps that occur from the moment `Statement.executeQuery()` is called until the resources are released. Detail why sequential retrieval is used and the architectural consequences of holding result sets open for too long.

**Guide answer:**
**1. Creation and Initial State:**
When `Statement.executeQuery(sql)` is executed, the database processes the query and returns a result stream. The JDBC driver creates a `ResultSet` object. The cursor is positioned at index `0` (before the first row). At this stage, no data can be read from the result set.

**2. Sequential Traversal & Cursor Movement:**
To read data, the client must call `next()`. The JDBC driver moves the cursor forward to the next row. The driver retrieves data on-demand or in chunks from the database stream. Data is read using getter methods mapping to the current cursor row (e.g. `rs.getString(1)`). This sequence continues in a loop until `next()` returns `false`, indicating the cursor has moved past the last row.

**3. Why Sequential Retrieval is Used:**
Sequential retrieval minimizes memory usage on the client side. Instead of loading millions of rows into Java heap memory at once, the database streams rows sequentially. The cursor-based model acts as a pointer to the current record, maintaining a lightweight footprint.

**4. Resource Clean-up and Architectural Consequences of Leaks:**
When processing is complete, `ResultSet.close()` must be invoked. If result sets are left open:
- **Database Cursor Leakage:** The database allocates server-side cursors for each query. Open result sets prevent these cursors from closing. Eventually, the database server runs out of available cursors, blocking subsequent queries.
- **Connection Pinning:** In connection pools, a connection with an active, unclosed `ResultSet` might not be returned to the pool properly, causing connection exhaustion.
- **Memory Overhead:** Temporary caching buffers in both the driver and the database server remain active, wasting memory.

**Key points to include for Full Marks:**
- Cursor state transitions (before-first $\rightarrow$ valid row $\rightarrow$ after-last).
- Explanation of sequential streaming memory benefits.
- Analysis of consequences of unclosed resources (cursor exhaustion, connection pinning).

**Common mistakes to avoid:**
Omitting the server-side database cursor exhaustion concept and focusing only on client-side Java memory.

## Section 4 — Study Tips & Weak Areas

- **The Cursor Offset Pitfall:** Students frequently forget that the cursor starts at position `0` (before-first). Always write a `while(rs.next())` loop or call `rs.next()` before accessing values.
- **Single Active ResultSet Rule:** Remember that executing another query on the *same* `Statement` object will automatically close the current `ResultSet`. If you need to iterate through a result set and execute queries based on each row, you **must** use a second, distinct `Statement` (or `PreparedStatement`) instance.
- **Nested Try-Catch vs Try-with-Resources:** Manually closing database resources in a `finally` block requires nested null checks and try-catch blocks which are error-prone. Master the **try-with-resources** syntax, as it closes the resources in reverse order of declaration automatically and prevents silent errors.
